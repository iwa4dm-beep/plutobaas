## 2. Feature matrix

### 2.1 Auto-Connect Studio — ✅
- Route: `src/routes/dashboard.auto-connect.tsx` (1,466 lines, full wizard)
- Server fns: `autoconnect/ai-planner.functions.ts`, `autoconnect/db-wizard.functions.ts`
- Client analyzer: `src/lib/autoconnect/analyzer.ts` (JSZip-based, static parsing)
- E2E: `e2e/auto-connect-permissions.spec.ts`, `e2e/auto-connect-role-switching.spec.ts`, `e2e/sidebar-auto-connect.spec.ts`
- Static findings: no TODO / stub inside the analyze → plan → apply chain. Env-mapper
  writes `# TODO:` comments for unknown env keys — that is by design.
- Gap: end-to-end `analyze → plan → apply → verify` needs a real Laravel + React ZIP
  to be **live-verified**; static path is complete.

### 2.2 Development (`/dashboard/pluto-deploy` dev tab) — ⚠️
- Server fns: `pluto/vps-deployer.functions.ts` (7 handlers), `pluto/db-wizard`,
  `pluto/vps-health.functions.ts`.
- Local dev commands and remote SSH paths exist. Works only when the VPS envs
  (`PLUTO_VPS_HOST`, `PLUTO_VPS_SSH_KEY`, …) are set as secrets.
- Gap: without those secrets, buttons return an error toast — expected. Docs cover
  the setup (`docs/DEPLOYMENT.md`, `pluto-backend/deploy/`).

### 2.3 Auto-Development (`ai-planner` + `e2e-runner`) — ✅ static / ⚠️ live
- `ai-planner.functions.ts` uses Lovable AI Gateway (Gemini 2.5 Flash) with a
  heuristic fallback — no API key still yields a plan (deterministic).
- `e2e-runner.ts` drives the plan-apply cycle; cancel-token tests pass
  (`sse-cancel-sql.spec.ts`, `e2e-cancel.spec.ts`).
- Live-verified only against fixture inputs. A real project ZIP would prove the
  full loop.

### 2.4 Publish — ⚠️ (two layers, both work as designed)
- **Frontend publish (Lovable app):** handled by Lovable platform via
  `preview_ui--publish`. Not code in this repo. Verified: `plutobaas.lovable.app`
  is live.
- **Backend publish (Pluto BaaS on your VPS):** `vps-deployer.functions.ts`
  drives build → deploy → verify against SSH targets. Same secret gate as §2.2.
- No stubs or TODOs in the deploy chain.

### 2.5 Multi-Project deploy — ⚠️
- Multi-tenant model: `workspaces` table + `workspace-provisioner.functions.ts` +
  `WorkspaceSwitcher.tsx` + `RequireWorkspace.tsx` all wired.
- RLS policies for workspace scoping live in
  `pluto-backend/migrations/0029_workspaces_tokens.sql`, `0016_stats_workspaces.sql`.
- Practical status: you can create N workspaces, each with its own tokens /
  keys / deploy targets. Every dashboard page reads the active workspace via
  `workspace-context.tsx`.
- Gap: there is no single "spin up a brand-new tenant end-to-end" wizard yet —
  the flow is: create workspace → provision → deploy → wire domain. That is
  4 clicks, not 1. Not broken, just not one-shot.

### 2.6 Custom Domain / Subdomain — ⚠️ (documented, partly manual)
- Route `/dashboard/custom-domains` renders (899 lines, add-domain flow, DNS
  check, audit tab, `e2e/custom-domains.spec.ts`).
- Server-side: `pluto/domain-audit.ts`, `pluto/domain-test.ts` do live DNS +
  reachability checks.
- Docs: `docs/CUSTOM-DOMAIN-SETUP.md`, `docs/api/custom-domains-phase65.md`,
  `pluto-backend/deploy/subdomains/install-all.sh` (renders nginx + issues certs).
- **What still requires manual action:**
  1. Frontend (Lovable) custom domain — connect via Lovable UI → Project Settings
     → Domains. Not scriptable from this codebase (platform feature).
  2. Backend subdomain (`api.<tenant>.example.com`) — run
     `sudo bash pluto-backend/deploy/subdomains/install-all.sh` on the VPS with
     the correct env. That is the runbook; not automated by a dashboard button.

## 4. Multi-project deploy runbook (single-source-of-truth)

For every new project you want to run on top of this stack:

1. **Frontend (Lovable app)**
   - Publish the Lovable project → get `<slug>.lovable.app`.
   - Project Settings → Domains → add root + `www` (or subdomain).
   - Set A `@`, A `www` → `185.158.133.1`; TXT `_lovable` verification.
2. **Backend workspace (Pluto BaaS)**
   - `/dashboard/projects` → create workspace, note the workspace ID / tokens.
   - `/dashboard/pluto-deploy` → run "Provision" for the new workspace.
3. **Backend subdomain (Pluto API)**
   - On the VPS:
     ```
     export ROOT_DOMAIN=example.com
     export TENANT_SLUG=acme        # → api.acme.example.com
     sudo bash pluto-backend/deploy/subdomains/install-all.sh
     ```
   - Verify: `curl -I https://api.acme.example.com/health` → 200.
4. **Wire the frontend to that backend**
   - Frontend env / Cloud secrets:
     `VITE_PLUTO_URL=https://api.acme.example.com`,
     `VITE_PLUTO_ANON_KEY=<workspace anon key>`.
   - Redeploy the Lovable project.
5. **Verify**
   - Load the custom domain → sign in → check `/dashboard/backend-status`
     shows the workspace healthy.

Repeat steps 2–5 per tenant. There is no cap on how many projects/tenants.

## 5. Overall verdict

| Area | Status |
|---|---|
| Auto-Connect Studio (static analyze → plan → apply) | ✅ |
| Development / VPS SSH ops | ⚠️ (needs VPS secrets) |
| Auto-Development (AI planner + e2e-runner) | ✅ / ⚠️ live |
| Frontend Publish (Lovable) | ✅ |

| Custom Domain — frontend | ⚠️ (Lovable UI, manual DNS) |
| Custom Domain — backend | ⚠️ (nginx/certbot script, VPS-side) |

**Nothing is broken.** What is missing is (a) a single wizard that stitches
tenant → workspace → backend subdomain → frontend domain together, and
(b) real live-verification against your own Laravel/React project and your
own domain.
